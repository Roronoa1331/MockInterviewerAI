### 5.1.1.系统开发工作流
<p>

##### 想法与需求

**是什么？在哪用？**

- **作用**：系统开发的起始阶段，明确系统的价值和目标
- **应用场景**：
  - 新系统开发初期
  - 需求分析
  - 目标明确
- **做什么的？** 系统开发的起始阶段，明确系统的价值和目标。
- **在哪用？** 所有新系统开发的初期阶段。

**会遇到哪些问题？用什么解决？**

- **如何让相关开发人员理解系统的价值和目标？**
  - **问题**：需要让开发人员理解系统价值和目标
  - **解决方向**：
    - **给相关开发人员能讲清楚：**
      - **价值和目标：** 价值传达
        - 明确系统价值
        - 明确系统目标
        - 清晰传达给开发人员
      - **体验和感受：** 体验传达
        - 明确系统体验
        - 明确系统感受
        - 清晰传达给开发人员
    - 进行头脑风暴
    - 编写需求文档
    - 组织需求评审

- **如何传达系统的体验和感受？**
  - **问题**：需要传达系统的体验和感受
  - **解决方向**：
    - 使用原型演示
    - 使用参考案例
    - 使用体验描述
    - 组织体验评审

- **如何避免过度细化任务，保持灵活性？**
  - **问题**：需要避免过度细化任务
  - **解决方向**：
    - **注意：如果是跟游戏内容相关的东西，交给开发团队的最好是虚的内容（目标、体验等），而不是具体切割好的任务或功能。这样可以让开发团队有更多的发挥空间，也能更好地应对需求变化：** 灵活性保持
      - 提供目标和体验
      - 避免过度细化任务
      - 保持开发灵活性
      - 应对需求变化
    - 关注目标和体验
    - 避免过度细化
    - 保持灵活性

**要点和思考方向：**
- 想法与需求是系统开发的起点
- 如果有新系统的需要，可以先做个头脑风暴
- 给相关开发人员能讲清楚价值和目标、体验和感受
- 如果是跟游戏内容相关的东西，交给开发团队的最好是虚的内容（目标、体验等），而不是具体切割好的任务或功能

##### 内容设计

**是什么？在哪用？**

- **作用**：策划设计系统功能、UI布局、操作流程和配置数据
- **应用场景**：
  - 系统开发的策划设计阶段
  - 功能设计
  - UI设计
  - 数据设计
- **做什么的？** 策划设计系统功能、UI布局、操作流程和配置数据。
- **在哪用？** 系统开发的策划设计阶段。

**会遇到哪些问题？用什么解决？**

- **如何清晰地表达系统功能？**
  - **问题**：需要清晰地表达系统功能
  - **解决方向**：
    - **策划交付：**
      - **系统功能说明：** 功能文档
        - **读者：程序、QA：** 文档读者
          - 明确文档读者
          - 针对读者编写
        - **需要清晰描述系统功能、交互逻辑、边界条件：** 功能描述
          - 描述系统功能
          - 描述交互逻辑
          - 描述边界条件
    - 编写系统功能说明
    - 使用图表辅助说明
    - 组织功能评审

- **如何设计UI布局和操作流程？**
  - **问题**：需要设计UI布局和操作流程
  - **解决方向**：
    - **策划交付：**
      - **UI设计：** UI设计文档
        - **Layout（图）设计：页面上有什么，放在哪：** 布局设计
          - **最佳实践：**
            - **策划能直接在游戏引擎中搭建UI布局、放置UI部件（用占位图）。这个过程可能会需要程序的支持协助：** 引擎布局
              - 在引擎中搭建布局
              - 使用占位图
              - 程序协助支持
            - **如果策划不能直接使用游戏引擎做UI Layout，就需要在另外的原型工具中制作Layout，程序也需要在后续的流程中完成这个UI初始搭建的工作：** 原型工具
              - 使用原型工具
              - 程序完成搭建
        - **流程（图）设计：操作流程：** 流程设计
          - **读者：程序、QA：** 流程文档读者
            - 明确文档读者
            - 针对读者编写
          - 设计操作流程
          - 使用流程图
    - 设计UI布局
    - 设计操作流程
    - 组织UI评审

- **如何设计数据驱动的配置？**
  - **问题**：需要设计数据驱动的配置
  - **解决方向**：
    - **策划交付：**
      - **元数据（配置数据）设计：数据驱动开发，设计配置表结构：** 配置设计
        - 设计配置表结构
        - 实现数据驱动
        - 优化配置管理
    - 设计配置表结构
    - 实现数据驱动
    - 优化配置管理

- **如何确保各方理解一致？**
  - **问题**：需要确保各方理解一致
  - **解决方向**：
    - **沟通：策划设计系统并跟技术、美术沟通，确保各方理解一致：** 团队沟通
      - 组织设计评审
      - 进行技术沟通
      - 进行美术沟通
      - 确保理解一致
    - 组织设计评审
    - 进行团队沟通
    - 确保理解一致

**要点和思考方向：**
- 内容设计是系统开发的基础
- 清晰表达系统功能、UI布局和操作流程
- 设计数据驱动的配置
- 确保各方理解一致

##### 代码重构

**是什么？在哪用？**

- **作用**：在不改变代码外部行为的前提下，改善代码的内部结构，提高代码质量
- **应用场景**：
  - 当代码变得难以维护、理解或扩展时
  - 消除技术债务
  - 提高代码可读性和可维护性
  - 优化代码性能
  - 准备添加新功能前的代码整理
- **做什么的？** 在不改变代码外部行为的前提下，改善代码的内部结构，提高代码质量。
- **在哪用？** 当代码变得难以维护、理解或扩展时，需要进行重构。

**会遇到哪些问题？用什么解决？**

- **如何判断何时需要重构？**
  - **问题**：需要判断何时需要重构
  - **解决方向**：
    - 代码有"坏味道"（Code Smell）：长函数、重复代码、过大的类等
    - 添加新功能困难，需要修改多处代码
    - 代码难以理解，需要大量注释才能看懂
    - 测试难以编写，依赖关系复杂
    - 性能问题，需要优化代码结构

- **如何安全地进行重构？**
  - **问题**：需要安全地进行重构
  - **解决方向**：
    - **建议：**
      - **要有比较好的测试机制支撑，比如单元测试，自动测试等。否则重构的成本会很大：测试是重构的安全网，没有测试的重构风险很高：** 测试支撑
        - 建立测试机制
        - 使用单元测试
        - 使用自动测试
        - 降低重构成本
    - 建立完善的测试机制（单元测试、集成测试）
    - 小步重构，每次只做小的改动
    - 使用版本控制，便于回滚
    - 重构前先理解代码，不要盲目重构
    - 使用重构工具（IDE的重构功能）

- **如何避免重构引入新的bug？**
  - **问题**：需要避免重构引入新的bug
  - **解决方向**：
    - 重构前确保有足够的测试覆盖
    - 重构后运行所有测试
    - 使用自动化测试验证行为不变
    - 代码审查，让其他人检查重构
    - 逐步重构，每次验证后再继续

- **重构的具体方法**
  - **问题**：有哪些常用的重构方法？
  - **解决方向**：
    - **作用：**
      - **让代码变得更清晰、更稳定、更可维护：通过重构消除技术债务，提高代码质量：** 重构目标
        - 让代码更清晰
        - 让代码更稳定
        - 让代码更可维护
        - 消除技术债务
    - **方法举例：**
      - **提炼过长的函数：将长函数拆分成多个小函数，提高可读性：** 提炼函数
        - 拆分长函数
        - 提高可读性
      - **降低耦合：减少模块之间的依赖关系，提高代码的模块化程度：** 降低耦合

##### 常见问题

##### 网游 vs 主机

| 问题 | 解答 |
|------|------|
| 为什么主机游戏和PC网游、手机网游的里程碑不太一样？ | 因为网游可以在游戏上线后持续的更新、维护，但主机游戏的审查机制和迭代更新方式都不同，所以产品需要在发布之前尽量做的完善。 |

| 类型 | 特点 |
|------|------|
| **网游** | 可以持续迭代，上线后可以修复问题、添加内容，里程碑更注重快速验证和迭代 |
| **主机游戏** | 发布后更新困难，需要通过严格的审查，里程碑更注重完整性和稳定性 |

##### 自研 vs 外包

| 问题 | 解答 |
|------|------|
| 自研和外包有什么不同？ | 自研以产品和盈利为中心；外包以时间和质量为中心。 |

| 模式 | 特点 |
|------|------|
| **自研** | 以产品和盈利为中心：关注产品的市场表现和盈利能力；在有限成本内做出吸引人的产品：需要在成本控制下创造有竞争力的产品 |
| **外包** | 以时间和质量为中心：关注按时交付和交付质量；按时、按质的去完成客户的需求：严格按照合同要求完成开发任务 |

### 5.3.2.版本管理
<p>
在产品的开发和运营过程中，特别是对大的产品和项目，可能会有大量的分支、版本存在。有效的管理好版本，会让项目不陷入混乱，不影响整体效率。版本管理包括版本控制系统、分支管理、资源组织等方面。
</p>

**关键词:**<br/> 
*git,分支,版本控制,分支管理,资源管理,热更新,AI Coding*

*等级: 初级|中级, 阶段: 开发, 分类: 管理能力, 角色: 客户端开发|服务端开发|全栈开发|管理*

##### 解决的问题

| 问题 | 说明 |
|------|------|
| **做好分支的规划** | 合理规划分支结构，避免分支混乱 |
| **做好分支定期的合并工作** | 定期合并分支，避免长时间不合并导致合并困难；长时间不合并的分支将变得非常难以合并（分支差异越大，合并冲突越多，解决成本越高） |
| **处理好大文件的版本管理** | 游戏开发中经常有大文件（美术资源、音频等），需要特殊处理 |

##### 版本控制系统

| 维度 | 内容 |
|------|------|
| **作用** | 用于管理代码和资源的版本历史，支持团队协作和版本回溯 |
| **应用场景** | 所有需要版本管理的项目；多人协作的项目；需要版本回溯的项目；需要管理代码和资源的项目 |
| **做什么的？** | 用于管理代码和资源的版本历史，支持团队协作和版本回溯 |
| **在哪用？** | 所有需要版本管理的项目，特别是多人协作的项目 |

| 版本控制概念 | 说明 |
|--------------|------|
| **中心式** | 上传（submit）：提交到中心服务器；更新（update）：从中心服务器获取最新版本 |
| **分布式** | 提交（commit）：本地记录更改；推送（push）：推送到远程仓库；拉取（pull）：获取最新更改 |
| **分支（branch）** | 开分支让不同工作任务独立完成，避免互相影响 |
| **标签（tag）** | 版本发布时打tag，便于快速找到版本内容 |

##### 分支管理

| 维度 | 内容 |
|------|------|
| **作用** | 通过分支策略管理不同功能的开发和发布流程 |

##### 常见问题与解决方案

##### 一、结点验收标准的定义

| 维度 | 内容 |
|------|------|
| **作用** | 定义工作节点的验收标准，确保交付质量 |
| **应用场景** | 所有需要阶段性交付的项目；工作节点验收时；需要确保交付质量时 |
| **做什么的？** | 定义工作节点的验收标准，确保交付质量 |
| **在哪用？** | 所有需要阶段性交付的项目 |

| 问题 | 解决方向 |
|------|----------|
| **如何定义验收标准？** 任务导向问题：只关注任务完成，不关注实际效果；应以体验为导向、以用例为导向 | 以体验为导向定义标准；以用例为导向定义标准；团队共同审视；明确体验目标和用例效果 |
| **如何确保交付质量？** | 建立明确的验收标准；在阶段过程中持续审视；团队共同验收；及时反馈和改进 |
| **如何避免"完成了但不可用"的情况？** | 以体验和用例为导向；确保可见和可审查；在过程中持续审视；及时发现问题并解决 |

| 要点和思考方向 |
|----------------|
| 验收标准应该以体验为导向，以用例为导向 |
| 在阶段过程中团队共同审视是否达成体验目标和用例效果 |
| 避免只关注任务完成，不关注实际效果 |
| 确保交付内容可见、可审查、可用 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「验收标准」「体验导向」「用例导向」「可审查」 |
| **方法** | 让 AI 协助设计验收标准、体验目标、用例效果 |
| **应用** | 验收标准定义、交付质量、阶段审视 |
| **提示词范例** | 「工作节点需验收标准，避免只关注任务完成不关注效果，请协助设计以体验和用例为导向的验收标准模板，以及团队共同审视的检查要点」 |

##### 二、问题到很后期才去反应

| 维度 | 内容 |
|------|------|
| **作用** | 及时发现和解决问题，避免问题积累 |
| **应用场景** | 整个项目开发过程；需要持续监控质量时；需要避免问题积累时 |
| **做什么的？** | 及时发现和解决问题，避免问题积累 |
| **在哪用？** | 整个项目开发过程 |

| 问题 | 解决方向 |
|------|----------|
| **如何及时发现问题？** 进度导向问题：只关注进度不关注质量；应日常持续发现问题，在过程中解决 | 技术负责人在前期设计接入；日常开发进行Review；指出目标导向问题；建立问题发现机制 |
| **如何避免问题积累？** | 在过程中持续发现问题；及时解决问题；建立问题跟踪机制；定期审查问题 |
| **如何建立问题发现机制？** | 建立代码审查机制；建立设计审查机制；建立定期检查机制；建立问题反馈机制 |

| 要点和思考方向 |
|----------------|
| 日常持续发现问题，而非等到最后 |
| 技术负责人在前期设计就要接入，在日常开发就要Review |
| 指出团队没有以目标为导向的问题 |
| 把所有问题解决到过程之中，避免问题积累 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「问题发现」「Review」「设计接入」「目标导向」 |
| **方法** | 让 AI 协助设计问题发现机制、审查流程、跟踪机制 |
| **应用** | 问题发现、设计审查、日常 Review |
| **提示词范例** | 「需日常持续发现问题，技术负责人前期设计接入、日常 Review，请协助设计问题发现机制（设计审查、代码审查、定期检查），以及问题跟踪与解决流程」 |

##### 三、混淆需求和思路

| 维度 | 内容 |
|------|------|
| **作用** | 区分需求（要什么）和思路（怎么做），确保理解正确 |
| **应用场景** | 需求沟通时；任务分配时；需要明确需求时 |
| **做什么的？** | 区分需求（要什么）和思路（怎么做），确保理解正确 |
| **在哪用？** | 需求沟通和任务分配时 |

| 问题 | 解决方向 |
|------|----------|
| **如何区分需求和实现思路？** 混淆问题：将实现思路误认为是需求；应挖掘根本需求，关注体验和要解决的问题 | 区分"要什么"和"怎么做"；挖掘根本需求；关注体验和效果 |
| **如何挖掘真实需求？** 例：匹配系统要挖掘匹配的体验和效果，关注根本目的 | 询问"为什么"而非"怎么做"；关注体验和效果；理解根本目的；评估方案是否达成目的 |
| **如何避免按错误思路实现？** | 先明确真实需求；评估实现思路是否合适；与需求方确认理解；选择最佳实现方案 |

| 要点和思考方向 |
|----------------|
| 区分需求（要什么）和思路（怎么做） |
| 挖掘对方的根本需求，关注体验和要解决的问题 |
| 关注体验和效果，而非具体的实现方式 |
| 如果方案可以达成根本目的就采纳，否则挖掘真实目的 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「需求」「思路」「要什么」「怎么做」「根本需求」 |
| **方法** | 让 AI 协助设计需求挖掘问题、需求确认流程 |
| **应用** | 需求沟通、需求挖掘、方案评估 |
| **提示词范例** | 「需区分需求与实现思路，避免将「怎么做」误认为「要什么」，请协助设计需求挖掘问题（为什么、体验目标），以及评估方案是否达成根本目的的检查清单」 |

##### 四、过度追求"速度"

| 维度 | 内容 |
|------|------|
| **作用** | 平衡开发速度和代码质量，避免技术债务 |
| **应用场景** | 功能开发时；系统设计时；需要平衡速度和质量时 |
| **做什么的？** | 平衡开发速度和代码质量，避免技术债务 |
| **在哪用？** | 功能开发和系统设计时 |

| 问题 | 解决方向 |
|------|----------|
| **如何平衡速度和质量？** 速度导向问题：只关注速度忽略质量和可维护性；应以目标为导向，达成目标而非完成任务 | 明确任务目的和目标；关注可维护性和可扩展性；在过程中多跟进；平衡速度和质量 |
| **如何避免过度追求速度？** | 明确任务目标；关注系统质量；评估技术债务；建立质量检查机制 |
| **如何评估技术债务？** | 评估可维护性；评估可扩展性；评估重构成本；建立技术债务管理机制 |

| 要点和思考方向 |
|----------------|
| 避免过度追求速度，忽略质量和可维护性 |
| 在项目计划和功能设计时多参与、多沟通 |
| 明确任务的目的和目标，达成目标而非仅仅完成任务 |
| 关注系统的可维护性和可扩展性 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「速度」「质量」「技术债务」「可维护性」 |
| **方法** | 让 AI 协助设计质量检查、技术债务评估、平衡策略 |
| **应用** | 速度与质量平衡、技术债务管理、目标导向 |
| **提示词范例** | 「需平衡开发速度与代码质量，避免过度追求速度，请协助设计任务目标与验收标准、技术债务评估维度，以及过程中跟进的检查要点」 |

##### 五、寻找利益共同点

| 维度 | 内容 |
|------|------|
| **作用** | 找到公司和个人的共同利益，促进合作共赢 |
| **应用场景** | 团队管理时；激励团队成员时；需要促进合作时 |
| **做什么的？** | 找到公司和个人的共同利益，促进合作共赢 |
| **在哪用？** | 团队管理和激励时 |

| 共同利益示例 | 说明 |
|--------------|------|
| **赚钱的共同利益** | 项目做得更好，团队获得更多奖励，公司获得更丰厚利润 |
| **技术眼界的共同利益** | 项目更大更优质，个人技术成长，公司技术储备更扎实 |
| **成长的共同利益** | 公司成长，个人获得更大成长空间 |

| 问题 | 解决方向 |
|------|----------|
| **如何找到共同利益？** 对立关系问题：将公司和个人视为甲方乙方；应寻找共同利益点（通常即项目成功） | 寻找项目成功等共同利益点；成就公司目标和利益；成就个人发展和利益 |
| **如何平衡公司和个人利益？** | 寻找双赢的共同利益点；平衡短期和长期利益；建立激励机制 |
| **如何避免对立关系？** | 寻找共同利益点；建立合作共赢关系；避免甲方乙方思维；促进团队主观能动性 |

| 要点和思考方向 |
|----------------|
| 寻找公司和个人的共同利益点，通常就是项目的成功 |
| 作为管理者，一方面成就公司目标和利益，一方面成就个人发展和利益 |
| 项目成功、技术成长、共同成长等都是双赢的共同利益点 |
| 避免将公司和个人关系视为对立，促进合作共赢 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「共同利益」「激励」「双赢」「项目成功」 |
| **方法** | 让 AI 协助设计激励机制、共同利益挖掘、双赢策略 |
| **应用** | 激励机制、利益对齐、合作共赢 |
| **提示词范例** | 「需寻找公司与个人的共同利益点（项目成功、技术成长、共同成长），请协助设计共同利益识别框架、激励机制要点，以及避免甲方乙方对立的沟通策略」 |

##### 六、一致目标

| 维度 | 内容 |
|------|------|
| **作用** | 确保团队成员对目标有一致的理解 |
| **应用场景** | 任务分配时；目标设定时；需要团队协作时 |
| **做什么的？** | 确保团队成员对目标有一致的理解 |
| **在哪用？** | 任务分配和目标设定时 |

| 问题 | 解决方向 |
|------|----------|
| **如何确保目标一致？** 个人目标不一致导致效率低下；应挖掘真实目标并沟通清楚 | 挖掘每个阶段的真实目标；与每个相关人员沟通清楚；确保团队目标一致 |
| **如何沟通目标？** | 明确表述真实目标；与相关人员沟通；确认理解一致；定期回顾和调整 |
| **如何处理目标冲突？** | 挖掘真实目标；寻找共同目标；平衡不同目标；建立目标优先级 |

| 要点和思考方向 |
|----------------|
| 个人目标不一致会导致团队事倍功半 |
| 项目在不同阶段、不同条件下，每个事情的真实目标可能不同 |
| 需要挖掘出每个阶段每个事情的真实目标 |
| 与每个相关人员沟通清楚真实目标，确保团队目标一致 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「目标一致」「真实目标」「沟通」「目标冲突」 |
| **方法** | 让 AI 协助设计目标沟通、目标挖掘、对齐流程 |
| **应用** | 目标设定、目标沟通、目标对齐 |
| **提示词范例** | 「不同阶段真实目标可能不同，需与相关人员沟通清楚，请协助设计目标挖掘问题、目标沟通模板，以及确保团队目标一致的确认流程」 |

##### 七、"追责"这个事

| 维度 | 内容 |
|------|------|
| **作用** | 建立问题处理机制，避免过度追责导致的问题 |
| **应用场景** | 问题处理时；团队管理时；需要建立健康团队文化时 |
| **做什么的？** | 建立问题处理机制，避免过度追责导致的问题 |
| **在哪用？** | 问题处理和团队管理时 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理问题？** 追责问题导致甩锅和内斗；适度压力有益，过度压力有害；应优先解决问题而非追责 | 优先解决问题，而非追责；让相关人员一起讨论；关注当下解决和将来规避 |
| **如何避免甩锅？** 单纯指责和压力会导致甩锅或掩盖问题；应建立健康团队文化 | 避免过度追责；建立健康团队文化；关注问题解决而非责任追究；适度压力，避免过度压力 |
| **如何建立健康的团队文化？** | 优先解决问题，而非追责；适度压力，避免过度压力；建立问题处理机制；促进团队协作而非内斗 |

| 要点和思考方向 |
|----------------|
| 过度追责会导致甩锅和内斗 |
| 适度的压力可以形成动力，过度的压力是内部损耗的原动力 |
| 遇到问题时，第一信号不要是指责，而是优先解决问题 |
| 建立健康的团队文化，避免过度追责，促进团队协作 |

| **方法** | 让 AI 协助设计问题处理流程、团队文化、适度压力 |
| **应用** | 问题处理、追责边界、团队文化 |
| **提示词范例** | 「遇问题需优先解决而非追责，避免甩锅和内斗，请协助设计问题处理流程（先解决、再讨论、关注规避）、适度压力与过度压力的边界，以及健康团队文化的实践建议」 |

##### OKR框架

| 维度 | 内容 |
|------|------|
| **作用** | 目标与关键结果框架，用于设定和跟踪目标 |
| **应用场景** | 需要明确目标和衡量结果的场景；从个人到组织都可以使用；需要目标对齐时；需要跟踪目标达成时 |
| **做什么的？** | 目标与关键结果框架，用于设定和跟踪目标 |
| **在哪用？** | 需要明确目标和衡量结果的场景，从个人到组织都可以使用 |

| OKR三问 | 说明 |
|---------|------|
| **我想去哪里？** | 设定清晰、有挑战性的目标 |
| **我怎么知道我到达了那里？** | 设定可衡量的关键结果，用于判断目标是否达成 |
| **我怎么到达那里？** | 制定执行计划，持续跟踪和调整 |

| 问题 | 解决方向 |
|------|----------|
| **如何设定有效的目标？** | 设定清晰、有挑战性的目标；目标要可达成但有一定挑战；目标要明确和具体；目标要可衡量 |
| **如何衡量目标达成情况？** | 设定可衡量的关键结果；关键结果要具体和可量化；定期跟踪关键结果；根据关键结果判断目标达成情况 |
| **如何确保目标对齐？** | 确保组织目标和个人目标对齐；确保团队目标对齐；定期回顾和调整目标；持续跟踪和调整 |

| 要点和思考方向 |
|----------------|
| OKR是目标与关键结果框架 |
| 目标要清晰、有挑战性（我想去哪里？） |
| 关键结果要可衡量，用于判断目标是否达成（我怎么知道我到达了那里？） |
| 制定执行计划，持续跟踪和调整（我怎么到达那里？） |
| 确保目标对齐，从个人到组织 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「OKR」「目标」「关键结果」「对齐」「可衡量」 |
| **方法** | 让 AI 协助设计 OKR 模板、关键结果、对齐流程 |
| **应用** | 目标设定、关键结果、目标对齐 |
| **提示词范例** | 「团队需 OKR 框架，请协助设计 OKR 三问（去哪里/怎么知道到达/怎么到达）的填写模板、关键结果的可衡量标准，以及组织与个人目标对齐的回顾流程」 |
