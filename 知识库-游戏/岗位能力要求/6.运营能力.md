##### 域名解析


| 维度           | 内容                                            |
| ------------ | --------------------------------------------- |
| **作用**       | 将域名转换为IP地址的基础服务，是游戏服务访问的入口                    |
| **应用场景**     | 游戏服务器入口域名解析；多线路加速IP解析；移动端域名解析                 |
| **DNS**      | 将域名转换为IP地址的基础服务                               |
| **HTTP DNS** | 基于HTTP协议的域名解析服务，用于移动端，避免运营商Local DNS劫持和跨网访问问题 |



| 问题                                 | 解决方向                                                                           |
| ---------------------------------- | ------------------------------------------------------------------------------ |
| **如何避免域名劫持？** 常见问题：域名劫持、跨网访问慢、单点故障 | 使用HTTP DNS避免Local DNS劫持；支持多IP解析，提供多个加速线路；使用Anycast、Unicast技术；结合CDN Pop节点优化解析路径 |
| **如何提高解析稳定性？**                     | 使用Anycast、Unicast技术；支持多IP解析；使用付费DNS服务；结合CDN Pop节点                              |
| **如何优化跨网访问？**                      | 使用HTTP DNS；支持多IP解析；结合CDN Pop节点；优化解析路径                                          |



| 要点和思考方向                           |
| --------------------------------- |
| 更多线路意味着可以更精确判断用户网络质量              |
| 付费和免费DNS服务在稳定性和功能上有差异             |
| 一个产品可设定多个加速线路，提供多个IP解析时，前端可根据策略选择 |
| 使用HTTP DNS避免Local DNS劫持           |
| 结合CDN Pop节点优化解析路径                 |



| 类型        | AI Coding 指南                                                                     |
| --------- | -------------------------------------------------------------------------------- |
| **交互提示**  | 说明需求；如「DNS」「HTTP DNS」「域名劫持」「多IP」「Anycast」                                        |
| **方法**    | 让 AI 设计解析策略、多线路配置、劫持规避                                                           |
| **应用**    | 域名解析、多线路、移动端解析                                                                   |
| **提示词范例** | 「游戏需域名解析，移动端用 HTTP DNS 避免劫持，请设计多 IP 多线路解析策略、前端根据网络质量选择 IP 的逻辑，以及与 CDN Pop 的配合要点」 |


##### 全球加速-动态加速


| 维度       | 内容                                  |
| -------- | ----------------------------------- |
| **作用**   | 实现跨地区网络访问的动态加速，让游戏服务器支持跨区域用户访问      |
| **应用场景** | 游戏服务器需要支持全球用户访问；实时性要求高的游戏；跨区域网络访问优化 |



| 延迟要求        | 说明      |
| ----------- | ------- |
| **普通游戏**    | 200ms以下 |
| **实时性高的游戏** | 35ms以下  |



| 问题                          | 解决方向                                                            |
| --------------------------- | --------------------------------------------------------------- |
| **如何解决延迟问题？**               | 使用加速通道、线路解析、回源、边缘节点；支持HTTP、TCP/UDP等加速协议；与域名解析配合，为不同服务商解析到不同加速线路 |
| **如何优化成本？** 让服务器靠近用户占比更高的区域 | 让服务器靠近用户占比更高的区域；优化加速线路选择；合理配置加速资源                               |



| 要点和思考方向                        |
| ------------------------------ |
| 加速通道、线路解析、回源、边缘节点的配置和选择        |
| 与域名解析配合使用，实现更灵活的加速策略           |
| 候选服务商：腾讯云GAAP、阿里云、网宿、橙域网络等     |
| 延迟要求：普通游戏200ms以下，实时性高的游戏35ms以下 |
| 成本优化：让服务器靠近用户占比更高的区域           |



| 类型        | AI Coding 指南                                                          |
| --------- | --------------------------------------------------------------------- |
| **交互提示**  | 说明需求；如「全球加速」「GAAP」「延迟」「加速通道」「边缘节点」                                    |
| **方法**    | 让 AI 设计加速架构、线路选择、成本优化                                                 |
| **应用**    | 动态加速、跨区域访问、延迟优化                                                       |
| **提示词范例** | 「游戏需全球加速，实时性要求 35ms 以下，请设计加速通道与边缘节点配置、与域名解析的配合，以及用户区域占比与服务器部署的成本优化策略」 |


##### 静态加速-CDN


| 维度       | 内容                            |
| -------- | ----------------------------- |
| **作用**   | 实现游戏静态资源下载加速                  |
| **应用场景** | 游戏包分发；热更资源存储；游戏内动态下载数据；静态资源加速 |



| 问题                                         | 解决方向                                       |
| ------------------------------------------ | ------------------------------------------ |
| **如何避免用户访问旧资源？** 新上传数据未预热、更新文件未刷新导致用户访问旧资源 | 新上传的数据要预热；更新的文件要刷新；注意每日预热有上限额度，需合理规划       |
| **如何选择合适的CDN？**                            | CDN主要用于静态资源加速，不适合动态业务；考虑节点分布、预热和刷新功能、成本和性能 |



| 要点和思考方向               |
| --------------------- |
| CDN主要用于静态资源加速，不适合动态业务 |
| 预热和刷新是CDN使用中的关键操作     |
| 需要关注CDN的预热额度限制        |
| 新上传的数据要预热，更新的文件要刷新    |
| 合理规划预热，避免超出每日额度       |



| 类型        | AI Coding 指南                                               |
| --------- | ---------------------------------------------------------- |
| **交互提示**  | 说明需求；如「CDN」「预热」「刷新」「热更」「静态资源」                              |
| **方法**    | 让 AI 设计预热刷新流程、额度规划、资源分发                                    |
| **应用**    | 热更资源、游戏包分发、预热刷新                                            |
| **提示词范例** | 「游戏热更资源用 CDN，需避免用户访问旧资源，请设计新上传预热、更新刷新的流程，以及每日预热额度限制下的规划策略」 |


##### 网络线路排查


| 维度       | 内容                          |
| -------- | --------------------------- |
| **作用**   | 排查网络连接问题，定位故障点              |
| **应用场景** | 玩家无法连接；延迟高；掉线等网络问题排查；网络故障定位 |



| 排查工具         | 说明          |
| ------------ | ----------- |
| **dig**      | 检查域名解析是否正确  |
| **ping**     | 排查解析地址和连接质量 |
| **telnet**   | 排查端口是否开放    |
| **mtr**      | 排查线路路由问题    |
| **网络万用表APP** | 移动端排查工具     |
| **腾讯华佗诊断工具** | 专业诊断工具      |



| 检查路径（从外到内）     | 说明                     |
| -------------- | ---------------------- |
| 1. 玩家切换Wifi/4G | 切换网络环境                 |
| 2. 玩家自身网络      | 检查玩家网络是否有问题            |
| 3. 地址解析        | 玩家端解析的地址对不对？是否走的加速IP？  |
| 4. 加速服务        | 加速本身有没有问题？（需加速服务商协助检查） |
| 5. 服务器内部       | 性能问题？服务间访问是否走了外网？      |



| 产品网络路径   | 说明                                                         |
| -------- | ---------------------------------------------------------- |
| **国内网络** | 域名解析 → 高防（防DDOS攻击） → 负载均衡（隐藏服务） → 游戏服务                     |
| **全球网络** | 域名解析（DNS加速） → 高防（防DDOS攻击） → 全球加速（服务加速） → 负载均衡（隐藏服务） → 游戏服务 |



| 问题              | 解决方向                                            |
| --------------- | ----------------------------------------------- |
| **如何排查网络连接问题？** | 使用dig检查域名解析；使用ping排查连接质量；使用telnet排查端口；使用mtr排查路由 |
| **如何按照检查路径排查？** | 按照从外到内的路径排查；逐步定位故障点                             |
| **如何理解产品网络路径？** | 理解网络路径各环节；根据路径排查问题                              |



| 要点和思考方向                                  |
| ---------------------------------------- |
| 按照从外到内的检查路径排查：玩家网络 → 地址解析 → 加速服务 → 服务器内部 |
| 使用dig、ping、telnet、mtr等工具排查问题             |
| 使用网络万用表APP、腾讯华佗诊断工具等专业工具                 |
| 理解产品网络路径：国内网络和全球网络的不同路径                  |
| 逐步定位故障点，从外到内排查                           |



| 类型        | AI Coding 指南                                                           |
| --------- | ---------------------------------------------------------------------- |
| **交互提示**  | 说明需求；如「dig」「ping」「telnet」「mtr」「排查」「故障」                                 |
| **方法**    | 让 AI 设计排查流程、检查路径、脚本工具                                                  |
| **应用**    | 连接问题排查、故障定位、玩家问题诊断                                                     |
| **提示词范例** | 「玩家无法连接/延迟高，需按外到内排查：玩家网络→解析→加速→服务器，请设计 dig/ping/telnet/mtr 的排查脚本与检查清单」 |

##### 负载均衡（SLB）


| 维度       | 内容                                         |
| -------- | ------------------------------------------ |
| **作用**   | 统一服务入口；隐藏业务服务器（公网IP），降低被攻击可能性；轮询主机，解决高可用问题 |
| **应用场景** | 作为游戏服务端业务的总入口；需要统一服务入口时；需要提升服务可用性时         |



| 问题              | 解决方向                        |
| --------------- | --------------------------- |
| **如何实现统一服务入口？** | 配置负载均衡器；设置统一域名；配置路由规则       |
| **如何提升安全性？**    | 隐藏真实服务器IP；只暴露负载均衡器IP；配置安全规则 |
| **如何解决高可用问题？**  | 配置多个后端服务器；实现健康检查；自动故障转移     |



| 要点和思考方向              |
| -------------------- |
| 负载均衡是游戏服务架构中的重要组件    |
| 通过隐藏真实服务器IP，提升安全性    |
| 通过多主机轮询，提升服务可用性      |
| 作为游戏服务端业务的总入口，统一服务入口 |
| 配置健康检查和自动故障转移，确保高可用  |



| 类型        | AI Coding 指南                                             |
| --------- | -------------------------------------------------------- |
| **交互提示**  | 说明需求；如「SLB」「负载均衡」「高可用」「健康检查」「隐藏IP」                       |
| **方法**    | 让 AI 设计负载均衡配置、健康检查、故障转移                                  |
| **应用**    | 统一入口、高可用、安全隐藏                                            |
| **提示词范例** | 「游戏服务需负载均衡：统一入口、隐藏真实IP、多主机轮询，请设计 SLB 配置要点、健康检查与自动故障转移策略」 |


##### 云联网


| 维度       | 内容                                 |
| -------- | ---------------------------------- |
| **作用**   | 让云上私有网络间（VPC）、VPC与本地数据中心间（IDC）内网互联 |
| **应用场景** | 绕开故障线路；游戏加速；企业内网和云服务之间的连通；跨区域网络连接  |



| 问题                                                      | 解决方向                        |
| ------------------------------------------------------- | --------------------------- |
| **如何绕开故障线路？** 服务器在A点，B到A故障但C到A正常时，可在C和A间建立云联网，让B用户经C转至A | 建立云联网连接；配置路由规则；实现故障自动切换     |
| **如何实现游戏加速？** 玩家就近接入POP点，走内部通道到达服务器源站                   | 配置POP点；建立内部通道；优化路由路径        |
| **如何实现内网互联？**                                           | 建立VPC间连接；建立VPC与IDC连接；配置内网路由 |



| 要点和思考方向                   |
| ------------------------- |
| 云联网提供内网级别的网络连接，延迟和稳定性更好   |

##### DDos攻击

| 维度 | 内容 |
|------|------|
| **作用** | 用大规模流量攻击游戏服务器，导致服务无法访问 |
| **攻击方式** | 用大规模流量（几十G甚至几百上千G）访问主机，运营商为保网络稳定会封堵IP，业务停转，攻击者勒索 |
| **应用场景** | 游戏服务器攻击；业务勒索；竞争对手恶意攻击 |

| 问题 | 解决方向 |
|------|----------|
| **如何预防DDos攻击？** | 使用负载均衡、反向代理等方式隐藏真实IP；避免直接暴露服务器IP；建立安全防护体系 |
| **如何应对DDos攻击？** 不要接受勒索给钱 | 快速迁移环境，更换IP恢复服务；使用多链路攻击分流；使用高防IP服务抵御攻击 |
| **如何理解攻击者的行为模式？** 攻击者有产业模式，攻击有成本，会评估目标价值 | 理解攻击者的产业模式和成本；根据业务价值评估攻击风险；制定相应的防护策略 |

| 要点和思考方向 |
|----------------|
| DDos攻击是最常见的网络攻击方式之一 |
| 预防比应对更重要，隐藏真实IP是关键 |
| 高防服务是有效的解决方案，但成本较高 |
| 需要建立应急响应机制，能够快速应对攻击 |
| 不要接受勒索，这会助长攻击者气焰 |
| 理解攻击者的行为模式，制定相应策略 |
| 建立完善的防护和应急响应机制 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「DDoS」「高防IP」「负载均衡」「隐藏IP」「多链路」 |
| **方法** | 让 AI 设计防护架构、应急迁移、高防选型 |
| **应用** | DDoS 防护、IP 隐藏、应急恢复 |
| **提示词范例** | 「游戏需防 DDoS，请设计负载均衡与反向代理隐藏真实 IP 的架构、高防 IP 与多链路分流方案，以及攻击后快速迁移与更换 IP 的应急流程」 |

##### 内容合规

| 维度 | 内容 |
|------|------|
| **作用** | 确保产品内容符合各地区法律法规和政策要求 |
| **应用场景** | 产品在不同地区运营时的内容审核和合规 |

| 内容类型 | 审核范围 |
|----------|----------|
| **文本** | 聊天内容、游戏内文本等 |
| **图片** | 游戏画面、UI图片等 |
| **音频** | 游戏音效、语音等 |
| **视频** | 游戏视频、宣传视频等 |

| 地区 | 合规要求 |
|------|----------|
| **中国大陆** | 聊天敏感词过滤；规范游戏内名词使用；画面（如血的表现）需符合要求 |
| **美国** | 政治正确；避免种族歧视等政治不正确的内容 |

| 敏感内容审核范围 | 涉黄/涉政/暴恐/违禁/辱骂/广告/未成年/仇恨言论/个人信息保护/私下交易等 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理不同类型的内容？** | 建立不同类型内容的审核机制；使用内容审核工具和服务 |
| **如何满足不同地区的合规要求？** | 针对不同地区建立相应的合规策略；使用地区特定的内容审核规则 |
| **如何审核敏感内容？** | 建立完善的敏感内容审核机制；敏感词过滤、内容审核是重要的合规手段 |

| 要点和思考方向 |
|----------------|
| 内容合规是产品运营的基础要求，必须严格遵守 |
| 不同地区有不同的合规要求，需要针对性地处理 |
| 敏感词过滤、内容审核是重要的合规手段 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「内容审核」「敏感词」「涉黄涉政」「文本」「图片」「音频」 |
| **方法** | 让 AI 设计审核机制、敏感词库、地区规则 |
| **应用** | 内容审核、敏感词过滤、多地区合规 |
| **提示词范例** | 「内容合规需审核文本/图片/音频/视频，请设计敏感词过滤、涉黄涉政暴恐等审核机制，以及中国大陆与美国等地区的差异化合规策略」 |

##### 地方性政策

| 维度 | 内容 |
|------|------|

| **应用场景** | 产品在不同地区运营；合规规划；合规检查 |

| **法国** | 《阿维亚法》 |
| **美国** | 《儿童网上隐私保护法》、《儿童在线保护法》、《儿童互联网保护法》、《通信内容端正法》；加州（CCPA）：加州消费者隐私法案 |
| **德国** | 《网络执行法》 |
| **澳大利亚** | Violent Material Act |
| **日本** | 《青少年网络环境整治法》 |
| **新加坡** | 《防止网络虚假和操纵法案》、《广播法》、《远程赌博法》、《计算机滥用和网络安全方案》、《新闻法》、《内部安全法》、《煽动法》、《网络行为法》、《不良出版物法》 |

| 问题 | 解决方向 |
|------|----------|
| **如何应对不同地区的政策要求？** | 了解并遵守各地区相关政策；建立合规检查清单 |
| **如何应对GDPR要求？** | 遵守GDPR数据保护法规；用户数据需要存储在当地 |
| **如何应对儿童保护要求？** | 遵守各地区儿童保护法规；建立儿童保护机制 |

| 要点和思考方向 |
|----------------|
| 不同地区的政策要求差异很大，需要深入了解 |
| GDPR对数据保护要求严格，需要特别注意 |
| 儿童保护是很多地区的重点，需要严格遵守 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「GDPR」「CCPA」「儿童保护」「数据本地化」 |
| **方法** | 让 AI 设计合规 checklist、数据保护方案、政策适配 |
| **应用** | 多地区政策适配、数据保护、儿童保护 |
| **提示词范例** | 「产品多地区运营需应对 GDPR、CCPA、儿童保护等政策，请设计各地区合规检查清单、数据本地化与存储要求、儿童保护机制」 |
