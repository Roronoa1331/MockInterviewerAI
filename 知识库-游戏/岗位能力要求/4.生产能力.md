| **地牢/地图布局** | 布局算法、布局验证、布局优化 |

|----------------|

### 4.1.4.引擎的其它系统
<p>
除了渲染、场景、UI 等「看得见」的部分，引擎内部还有一批支撑性系统，负责资源管理、核心通用能力、跨平台封装以及与第三方 SDK 的衔接。这些系统决定了引擎和产品能否高效开发、稳定运行和长期演进。
</p>

##### 资源管理

| 维度 | 内容 |
|------|------|
| **作用** | 管理所有游戏数字内容在磁盘和运行时中的组织、加载与生命周期 |
| **应用场景** | 游戏资源加载、资源生命周期管理、资源打包和分发、资源依赖管理 |

| 核心认识 |
|----------|
| 所有游戏的数字内容，都要以数据（文件）的形式进行存储和使用 |
| 资源管理决定了项目的可维护性、加载性能和包体结构 |

| 问题 | 解决方向 |
|------|----------|
| **如何管理各种资源类型** 需要管理多种类型的游戏资源 | 3D 模型、贴图、材质、字体、骨骼、碰撞数据、地图/关卡数据；建立资源类型系统；实现统一资源接口 |
| **如何建立资源规范** 需要统一的资源命名和目录规范 | 统一的资源命名与目录规范；建立命名规范、目录结构；实现自动化工具 |
| **如何管理资源依赖关系** 资源之间存在复杂的依赖关系 | 资源依赖关系管理（模型引用的材质、贴图、动画等）；实现依赖分析、依赖加载；使用依赖图管理 |
| **如何实现资源打包** 需要支持多种打包格式 | 支持 AssetBundle/Pak/自定义包格式等；实现增量打包；优化包体大小 |

| 典型资源类型 | 说明 |
|--------------|------|
| **3D 模型** | 模型加载、优化、格式管理 |
| **贴图** | 贴图加载、压缩、格式管理 |
| **材质** | 材质加载、参数管理、材质实例化 |
| **字体** | 字体加载、缓存、动态字体 |
| **骨骼** | 骨骼加载、骨骼动画管理 |
| **碰撞数据** | 碰撞数据加载、碰撞体类型管理 |
| **地图/关卡数据** | 关卡加载、关卡数据管理、流式加载 |

| 要点和思考方向 |
|----------------|
| 资源管理是游戏开发的基础 |
| 建立统一的资源规范提高可维护性 |
| 优化资源加载性能影响游戏体验 |
| 管理资源依赖关系避免加载错误 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明引擎、需求；如「AssetBundle」「Addressables」「依赖」「打包」「热更新」 |
| **方法** | 让 AI 设计资源规范、依赖管理、打包流程 |
| **应用** | 资源架构、打包策略、热更新 |
| **提示词范例** | 「Unity 资源需 AssetBundle 或 Addressables，请设计资源命名规范、目录结构、依赖分析流程，以及增量打包与热更新的实现要点」 |

##### 核心系统

| 实现完善的测试框架保证代码质量 |

##### 跨平台接口

| 维度 | 内容 |
|------|------|
| **作用** | 对操作系统和底层库进行封装，向上提供统一接口，让引擎逻辑尽量不关心具体平台差异 |
| **应用场景** | 多平台游戏开发、跨平台发布、底层库封装 |

| 问题 | 解决方向 |
|------|----------|
| **如何封装不同平台的功能** 需要封装不同平台的底层功能 | 文件系统、网络、容器/线程库、图形接口、物理接口；设计抽象层架构；实现平台适配层 |
| **如何实现接口统一** 需要让上层模块只依赖统一接口 | 让上层模块只依赖统一接口，可在项目后期更换/升级底层实现；设计统一接口；支持接口替换 |
| **如何简化多平台发布** 需要简化多平台发布流程 | 简化 PC/主机/移动端的差异处理；实现平台抽象、构建系统、自动化发布 |

| 典型封装方向 | 说明 |
|--------------|------|
| **文件系统** | 统一文件接口、异步 IO |
| **网络** | 统一网络接口、多协议支持 |
| **容器/线程库** | 统一容器接口、统一线程接口 |
| **图形接口** | 统一 DirectX/OpenGL/Vulkan/Metal 等 |
| **物理接口** | 对不同物理 SDK 的统一抽象 |

| 要点和思考方向 |
|----------------|
| 跨平台接口是引擎可移植性的基础 |
| 设计良好的抽象层提高可维护性 |
| 优化跨平台性能影响游戏体验 |
| 简化多平台发布流程提高开发效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明平台、需求；如「文件」「网络」「线程」「图形」「物理」「抽象层」 |
| **方法** | 让 AI 设计平台抽象层、接口统一、构建系统 |
| **应用** | 跨平台封装、接口设计、多平台构建 |
| **提示词范例** | 「引擎需跨平台：文件、网络、线程，请设计抽象接口与平台实现分离的 C# 架构，以及 PC/Android/iOS 的差异化处理策略」 |

| **应用场景** | 特定游戏类型开发、游戏特色功能实现、游戏玩法系统 |

| 问题 | 解决方向 |

| 基于引擎通用能力实现 |

##### 第三方接口 / SDK

| 维度 | 内容 |
|------|------|
| **作用** | 在操作系统之上，利用成熟的第三方库实现图形、物理、动画等基础功能服务 |
| **应用场景** | 图形渲染、物理模拟、动画处理、其他基础功能 |

| 问题 | 解决方向 |
|------|----------|
| **如何选择合适的 SDK** 需要选择合适的第三方 SDK | 评估性能、稳定性、许可协议、平台支持、社区活跃度 |
| **如何封装第三方 SDK** 需要合理封装第三方 SDK | 通过合理的封装，让第三方 SDK 可被安全升级或替换；实现抽象层；封装 SDK 接口 |
| **如何选择 SDK** 需要综合考虑多个因素选择 SDK | 综合考虑：性能、稳定性、许可协议、平台支持度与社区活跃度 |

| 类型 | SDK | 说明 |
|------|-----|------|
| **图形接口** | DirectX | Windows 平台图形，高性能渲染 |
| **图形接口** | OpenGL | 跨平台图形，广泛支持 |
| **图形接口** | Vulkan | 低开销 API，多线程支持，高性能 |
| **物理 SDK** | PhysX | NVIDIA 物理引擎，高性能 |
| **物理 SDK** | Havok | 专业物理引擎，商业授权 |
| **物理 SDK** | ODE | 开源物理引擎，免费使用 |
| **动画 SDK** | Granny | 专业动画引擎，商业授权 |

| 要点和思考方向 |
|----------------|
| 第三方 SDK 提供成熟的基础功能 |
| 合理封装提高可维护性和可替换性 |
| 综合考虑多个因素选择合适 SDK |
| 建立 SDK 管理规范提高开发效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「PhysX」「Havok」「DirectX」「Vulkan」「封装」「替换」 |
| **方法** | 让 AI 对比 SDK、设计封装层、给出选型建议 |
| **应用** | SDK 选型、封装设计、接口抽象 |
| **提示词范例** | 「游戏需物理引擎，请对比 PhysX、Havok、Bullet 的性能、许可、平台支持，并设计 C# 物理接口抽象层以便未来替换实现」 |

##### 设计原则

##### 所见即所得

| 维度 | 内容 |
|------|------|
| **作用** | 让策划、美术在编辑器中看到的效果，与游戏中真实效果尽可能一致，减少反复调试和来回切换 |
| **应用场景** | 游戏内容制作、美术资源制作、关卡设计、特效制作 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现所见即所得** 需要让编辑器预览效果与游戏运行时一致 | 预览视图使用与运行时一致的渲染管线与资源；关键参数修改后能实时反馈；必要时提供「运行时模拟」功能（播放特效、物理模拟）；使用相同的渲染系统；实现实时预览机制 |

| 要点与思考方向 | 说明 |
|----------------|------|
| **渲染一致性** | 使用相同的渲染管线、资源格式；实现实时预览 |
| **实时反馈** | 参数修改后能实时反馈；实现参数实时更新、预览刷新 |
| **运行时模拟** | 播放特效、物理模拟、动画播放；提供完整预览 |

| 要点和思考方向 |
|----------------|
| 所见即所得是编辑器设计的核心原则 |
| 使用与运行时一致的渲染管线保证效果一致 |
| 实现实时反馈提高编辑效率 |
| 提供运行时模拟功能增强预览能力 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明编辑器类型、需求；如「预览」「渲染一致性」「实时反馈」「运行时模拟」 |
| **方法** | 让 AI 设计预览架构、参数联动、渲染管线复用 |
| **应用** | 预览系统设计、实时反馈、效果一致性 |
| **提示词范例** | 「特效编辑器需所见即所得，请设计预览视图与游戏运行时共用渲染管线的 C# 架构，以及参数修改后实时刷新的机制」 |

### 4.3.3.DevOps
<p>
在游戏开发过程中，要不断地出数据和产品的版本来进行测试、调试和发布，这些过程如果全部依赖人工操作是非常低效且容易出错的。
通过建设 DevOps 体系（CI/CD、版本生产线、工程管理等），可以用自动化工具链和规范化流程，大幅提升项目的生产效率和稳定性。
</p>
